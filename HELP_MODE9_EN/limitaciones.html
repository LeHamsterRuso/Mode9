Library limitations

Mode 9 is an engine designed to run on modest hardware, so it has fixed limits on the amount of geometry, textures and objects it can handle at once. It is worth knowing these before designing levels so there are no surprises. If you exceed any of them, the excess is simply not drawn or not loaded.

Scene (LOAD_WLD9)

ResourceMaximumNotes
Vertices4096Points in the scene mesh.
Triangles (faces)8192 Total scene faces. Although 8192 fit, the fewer the better for performance.
Materials16Different textures per scene (usemtl 0..15).
Texture per material1024 x 1024 pxDIV .MAP files.
Collision walls4096 Vertical faces that stop the player. A subset of the total triangles.
Collision floors4096 Near-horizontal faces that support the player (ramps and steps).

Object models (LOAD_OBJ9 / LOAD_OBJ9_ANIM)

ResourceMaximumNotes
Distinct models at once8 Simultaneously loaded models. A complete animation, with all its poses, takes 1 slot (not 1 per pose).
Instances per frame32 How many times you can call DRAW_OBJ9() or DRAW_OBJ9_ANIM() in one frame across all models combined.
Vertices per model512 Per pose. In an animation, every pose shares this same cap (they all have the same vertex count).
Triangles per model1024 
Poses per animation (nframes)no fixed cap There is no maximum baked into the DLL: memory and the file name are the limits. Each pose stores only its vertex positions (512 × 12 bytes ≈ 6 KB per pose), so even a long animation uses little.
About the number of poses in an animation: because the files are numbered (CAT0.OBJ, CAT1.OBJ...), the name is still bound by the DOS 8-character limit. With a 6-letter base you only have room for 2 digits (up to 99 poses); with a short one like "CAT" you can reach thousands. In practice you never need that many: mode 9 blends the in-between poses for you, and 8–12 poses already give a very smooth cycle (see DRAW_OBJ9_ANIM()).

ResourceMaximumNotes

Viewports (split-screen)4 Simultaneous cameras with MODE9_SPLIT().
Regions16 Screen areas with MODE9_REGION().
Panoramic sky4096 x 2048 px .MAP sky texture (LOAD_SKY9()).

Tips for working around the limits

Too many scene polygons: let textures provide the detail instead of geometry (a brick wall is one quad with a good texture, not a thousand modelled bricks). And if a level is huge, split it into smaller maps and load the next one with LOAD_WLD9() when the player changes area: since loading is hot, it feels like a seamless world.

More than 16 materials: pack several small textures into a single 1024x1024 .MAP (a "texture atlas") and use UVs to pick each face's portion. One material then does the work of many.

More than 8 3D models or more than 32 instances per frame: reserve 3D models for what matters (the protagonist, nearby enemies) and do the rest with billboards, which support up to 256. The same model can also be reused with different textures to add variety without spending extra slots.

Models that are too dense: simplify the mesh before exporting (in Blender, the Decimate modifier). The engine is low-resolution and a very detailed model does not look better — it just runs slower.

If the game runs slowly: not a hard limit, but it helps to cut the view distance with SET_FOG9() or MODE9_FARCLIP() and lower the render resolution with MODE9_DOWNSCALE().

Note for programmers: these limits are fixed in the MODE9.DLL source code (constants MAX_TRIS, MAX_OBJ9_MODELS, etc.). Whoever compiles the DLL can raise them, at the cost of higher memory usage.
